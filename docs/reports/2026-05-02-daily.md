## 今日最大的事：Linux 内核安全披露流程的系统性失败
**来源：** Hacker News | **HN热度：** 584p / 527c | **日期：** 2026-04-30  
原文：https://www.openwall.com/lists/oss-security/2026/04/30/10

### 是什么
Gentoo 开发者 Sam James 披露了一个被忽视的事实：Linux 内核安全漏洞**不会自动通知发行版**。除非漏洞报告者主动选择将漏洞同时提交给 linux-distros 邮件列表，否则各大 Linux 发行版不会提前获得预警。

这正是 CVE-2026-31431（"Copy Fail"）暴露的问题——一个自 2017 年引入的高危本地提权漏洞（CVSS 7.8），允许任何本地用户通过 732 字节 Python 脚本在数秒内获得 root 权限。当漏洞公开时，绝大多数主流发行版尚未准备好补丁。

### 为什么热
1. **漏洞严重性**：Copy Fail 被比作 Dirty Cow/Dirty Pipe 级别，但更致命——无需竞争条件、100% 可靠利用、跨容器攻击
2. **披露流程的系统性失败**：内核团队在 4/1 提交补丁，但直到 4/29 公开披露时，Debian Stable、Ubuntu LTS、RHEL 等主要发行版仍未发布修复版本
3. **理念冲突**：「负责任的披露」vs 「立即公开」的激烈争论

### 各方声音
**支持立即公开：**
- "Immediate public disclosure is the only choice that isn't irresponsible" — HN 用户
- "Two things can be true: the kernel ecosystem should have done better, and publicly sharing the exploit was irresponsible" — HN 用户

**支持现行流程：**
- "The kernel devs patched the kernel. Distro maintainers can see kernel changes. It is literally not the vulnerability researcher's problem" — HN 用户

**安全厂商：**
- "Copy Fail is a reliable local privilege escalation that turns an eight-year-old performance optimization into a four-syscall path from unprivileged user to root" — Sysdig 威胁研究团队

### 背景上下文
- 内核文档曾鼓励报告者向 linux-distros 披露，但在争议后修改为「强烈建议不要同时联系两个列表」
- Copy Fail 是使用 AI 辅助扫描工具（Xint Code）发现的——仅用约 1 小时扫描就找到存在 8 年的漏洞
- AI 驱动的漏洞发现已成现实，未来类似漏洞可能以更快速度被发现和披露

### 【判断】
1. **基础设施安全策略需重新审视**：默认信任 Linux 内核隔离边界可能是危险的，容器边界可以被轻易跨越
2. **「负责任的披露」没有行业标准**：不要假设上游项目会按你认为「合理」的方式披露漏洞
3. **建议行动**：立即检查内核版本（`uname -r`），如果在 4.14-7.0-rc 范围内，优先应用临时缓解措施
